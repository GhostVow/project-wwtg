## 根因分析

### 1. POI 分类覆盖不足（主因）

当前 `amap_config.py` 只有 6 个 type code，偏文旅/寺庙：

```python
AMAP_TYPE_CODES = {
    "景点": "110000",
    "餐饮": "050000",
    "咖啡": "050500",
    "亲子": "141200|141300",
    "休闲": "080000",
    "博物馆": "110201",
}
```

**缺少：** 公园/广场、体育运动、商业街/购物、电影院/娱乐。LLM 巧妇难为无米之炊——POI 库里全是古寺/景区，推荐结果自然偏向文化类。

### 2. LLM 模糊意图识别不足（次因）

- "两个人" → 未推断出情侣/朋友属性区别
- "想动一动" → 未提取出运动意图
- "和男朋友" → ✅ 正确识别情侣
- "想运动跑步" → ✅ 正确推运动类 POI

意图识别高度依赖关键词，对模糊/隐含表达处理不够。

---

## 改进建议

### 高优先级

1. **扩展 POI 分类**（成本低、效果大）

```python
# 新增到 AMAP_TYPE_CODES
"公园": "110101|110102",   # 公园/广场
"体育": "080100|080200",   # 体育场馆/健身中心
"购物": "060100|060400",   # 商场/步行街
```

预估新增 27 次 AMAP 调用/天（3类 × 3城 × 3页），远在 5000/天限额内。

2. **增加澄清提问机制** — 当用户意图模糊时，系统主动追问：
   - "是情侣还是朋友出行？"
   - "想轻度散步还是出汗运动？"
   - 比靠关键词猜更可靠

### 中优先级

3. **应用层 POI 预筛** — 根据场景标签筛选 top 30-50 POI 再传 LLM，减少 token、加快响应
4. **Plan 数据持久化** — 从内存字典迁移到 Redis/DB，容器重启不丢失
