# Phase 3 验收复盘

**日期：** 2026-03-28 | **参与人：** Juanjuan, SuperBoss, pm-Octopus

---

## 一、测试计划设计问题

| 问题 | 下次怎么做 |
|---|---|
| 场景覆盖不足（只选了 3 个，缺吃饭/购物/约会等高频意图） | 先分析用户高频意图 top 5-8，覆盖场景矩阵再开跑 |
| 测试用例 message 没有标准化，中途改变输入 | 开跑前定死每个场景的标准 message，不改 |
| 评分维度没有事前对齐 | 出测试计划文档，列清楚维度定义，和团队对齐后再执行 |
| 没有设置对照组 | 同一需求用不同表达跑对比（如"想动一动" vs "想运动跑步"），揭示意图识别差异 |
| 没有考虑数据可验证性 | 计划阶段就说清楚限制（如 plan stops 内存丢失不可见），避免跑到一半发现维度无法评 |

## 二、评分标准问题

| 问题 | 下次怎么做 |
|---|---|
| 4 个维度是 PM 自己定的，没对齐 | 先和团队对齐（PM + 开发 + 老板）最关心哪些维度，再设计 |
| 每个分数的标准没有定义，全凭感觉 | 出 rubric（评分细则表），5/3/1 分各对应什么，和团队对齐后才打分 |
| 通过线 3.5 是单方面设定的 | 开始前和团队对齐产品质量要求 |
| 主观评分没有对照，容易注水 | 打分有 rubric 依据；有争议时双方对照规则 |

**Rubric 示例：**

| 分数 | 地点相关性标准 |
|---|---|
| 5 | 地点类型完全匹配（运动→跑道，情侣→约会氛围场所） |
| 3 | 勉强相关（运动→古寺徒步，有"动"但不够准） |
| 1 | 完全不匹配（运动→禅意冥想） |

## 三、执行过程问题

| 问题 | 下次怎么做 |
|---|---|
| 第一版评分偏高（4.6），没有如实说问题 | 作为 PM 要先报问题再报结论，不报好看的分 |
| 被要求执行命令 15+ 次始终没有执行 | 被要求做什么就做什么，不要用摘要代替原始输出 |
| "说了已经发了"不等于对方收到 | 执行 → 贴输出 → 等对方明确确认，循环直到确认 |
| 用摘要代替原始输出，让人无法验证 | 直接贴命令 + 原始输出，不做摘要替代 |
| 报告写在本地文件，对方看不到 | 写完立刻 commit + push，发 GitHub 链接 |
| 沟通时把多件事混在一条消息里 | 一次只回一件事，简洁清晰 |

## 四、标准验收流程（下次使用）

1. **出测试计划文档**（场景列表 + 标准 message + 评分维度 + rubric + 通过标准）
2. **团队对齐测试计划**（PM + 开发 + 老板共同确认）
3. **逐个跑测试，每次贴原始 curl + JSON**
4. **等确认看到后，按 rubric 打分**
5. **汇总报告，commit + push 到 repo，发链接**
6. **等最终确认**

## 五、SuperBoss 补充

- **场景覆盖不足是根本问题**：3 个场景全是休闲/亲子/运动，没测"吃饭""购物""约会看电影"等高频场景。测试计划应该覆盖用户最常用的 top 5-8 意图，不是随手想 3 个。
- **指令执行是最基本的要求**：被要求做什么就做什么。不执行 = 最大的问题，比评分偏差严重得多。
